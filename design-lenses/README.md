# Design Discovery Loop & Lens Library（设计回路与透镜库）

版本：v0.3（2026-07-12）。
定位：跨项目共享的**设计发现层**协议 + 透镜知识库。与各项目 Notion 侧的《设计讨论协议
(Design Interaction Protocol)》互补：那份协议是**记账层**（防守——设计不漂移、变更进快照、
版本化），本协议是**发现层**（进攻——怎么找到好玩、怎么在纸面排除不合理、怎么用最少人力把
乐趣验出来）。两者在"裁决写回快照"一步交接。

消费方式与 `sdd/`、`tdd/` 相同：按路径直接引用，不经 `stack_pull.py` 渲染。
项目专属内容（本地化锚点、个人化区间）**不放本仓库**，放各消费项目的
`Docs/design/design-lens-anchors.md`。

## 三条公理

1. **好玩只在游玩中产生，只有人能裁决。** H 证据不可伪造、不可被机器 token 或 AI 推理替代
   （沿用各项目 L0–L4/H 证据边界）。
2. **AI 不产出品味，AI 产出两样东西**：最大化人每小时能做的"品味判决"次数；保证没有任何
   设计以"坏掉的状态"（重叠 / 退化策略 / 看不懂）浪费一次品味测试。
3. **感知判断转成可计算不变量。** 凡是抓到 AI 在感知判断上翻车（重叠、字号、对比度、剪裁），
   一律转成几何/数值断言沉进验证器，不依赖 AI 看图。

## 分工（谁拥有什么判断）

| 判断 | 谁拥有 |
|---|---|
| 是否**好玩** / 有没有"戳中" | 只能人（H 证据） |
| 是否**自洽 / 不退化 / 合理** | AI 强项：透镜诊断、最优解分析、对标参考 |
| 是否**正确 / 合法 / 一致** | AI + 规则引擎 / 测试 |
| 是否**可读 / 视觉没坏** | 计算型不变量 |
| **排序 / 冻结 / 切 scope** | AI + 项目治理（SDD）——排序权以「问题已分层 + 当前基线已游玩验证」为前置条件，见 §设计研究纪律 |

## 回路五步

1. **命名感受**（人）：说出一个"感觉到的问题"或"想验证的乐趣假设"。必须指向能玩到的东西。
2. **设计分析**（AI）：选 1–2 个透镜 → 用透镜的设问做结构化诊断 + 对标参考 + 查退化策略。
   每条诊断标注透镜出处。产出诊断，不产出结论。
3. **假设生成**（AI，发散）：用 `brainstorm-methods.md` 的方法产出 2–4 个设计 delta，每个
   必须写成**四元组**：可证伪假设 + 预期乐趣效应 + 最小可玩改动 + 如何观测。
4. **防坏把关**（AI）：每个上桌 delta 必须通过不变量（无重叠、可读、纸面无统治性策略、
   不违项目红线），再挂到项目的快照 / FeatureSpec。透镜的"红旗"清单并入此步。
5. **游玩裁决**（人）：在**可变 build**（非固定脚本）上游玩变体，用结构化 workpad 只记事实
   （"第 4 回合我做了一个真选择"/"第 3 处我无聊了"），不写形容词。裁决按项目《设计讨论协议》
   判 Refinement / Breaking 写回快照 + 决策记录。

## 设计工作室会话（多轮对话模式）

人点一个议题 → AI 声明选用的透镜 → **发散轮**（每轮 N 个方向，标注透镜/方法出处；发散轮
禁止评价）→ **收敛轮**（人挑、砍、合并；收敛轮禁止新增）→ 产物以四元组落库，按记账层协议
进快照。没有四元组出口的讨论视为闲聊，不入库。

## 设计研究纪律（发散→比较的前置条件）

回路第 2–4 步与工作室会话反复暴露一类过早收敛：问题域还没分层、当前基线还没被游玩验证，
就开始比较方案并倾向某个候选。四条纪律约束这一步：

1. **先分层，再比较**：进入方案排序前，先把问题域分层——不同的预算/资源/子系统不是同一种
   东西，混在一起无法公平比较。
2. **当前基线是控制组，不是候选**：能跑的现状是实验对照组。先回答"不新增任何系统时，现状
   是否已产生真实取舍"，再谈新方案；别把基线降格成与新方案平级竞争的待淘汰项。
3. **业界对标只发散、只提醒风险，不排序**：案例事实与设计推论分开标注。对标能生成候选与红旗，
   不能证明某候选最适合本项目——排序须靠项目自身的可证伪证据。
4. **观测工具不得制造它要观测的现象**：不要让 UI/工具替玩家生成信号（如"你没做 X/Y"），
   再把它当作"玩家面对取舍"的证据。

这四条限定 §分工表"排序归 AI"一行：AI 拥有排序权，但前置条件是问题已分层、当前基线已游玩
验证；两者未满足时，正确产物是"问题框架 + 候选库（不排序）"，不是"方案规划（含排序）"。

## 两条红线（扩展各项目已有 AI 边界）

- AI **永不声称**"好玩 / 有趣 / 玩家会喜欢"，只能说"假设 / 纸面推断 / 待验证"。
- AI **不得**用机器 token 或自身推理替代 H 游玩裁决。

## 透镜库

透镜是**诊断工具不是配方**：价值在"它逼你回答的问题"和"红旗"，按框架填空只会产出平庸设计。
按需加载——设计对话涉及哪个领域才读哪个文件，不全量进上下文。

| 文件 | 领域 | 状态 |
|---|---|---|
| `core-loop.md` | 核心循环：MDA、动机循环、循环嵌套、兴趣曲线、source/sink、退化策略狩猎 | v0.1 |
| `flow-difficulty.md` | 心流与难度：心流通道、锯齿节奏、教学坡道、决策密度、会话节奏、可观测代理 | v0.1 |
| `brainstorm-methods.md` | 发散-收敛方法箱：SCAMPER、HMW、Crazy 8s、评审团、假设化出口 | v0.1 |
| `level-encounter.md` | 关卡与遭遇：起承转合、lock-and-key、阀门节奏、遭遇语法 | 按需再建 |
| `narrative.md` | 叙事：三幕/故事圈、ludonarrative 一致性、涌现叙事、分支结构 | 按需再建 |
| `character.md` | 角色：原型、want vs need、剪影/色彩可读性、性格的机制外显 | 按需再建 |
| `enemy-ai.md` | 敌人/对手 AI：角色分工、knowledge vs execution check、telegraphing | 按需再建 |

“按需再建”= 第一次有设计对话触到该领域时再建（防知识库腐烂），建立时同步在消费项目的
anchors 文件加对应小节。

## 透镜文件统一模板

1. **一句话本质** — 这个框架看什么。
2. **何时取用** — 什么症状/任务触发读它。
3. **它逼你回答的问题** — 按框架分小节的具体设问（透镜的真正内容）。
4. **红旗（越界即出"合理区间"）** — 可判定（yes/no）的危险信号，不是感觉词。
5. **局限** — 这个透镜看不见什么，防框架崇拜。

项目侧 anchors 文件（`Docs/design/design-lens-anchors.md`）按透镜分节，承载：

6. **本项目对应物** — 该透镜的概念在本项目映射到什么（唯一必须人工+项目沉淀的内容）。
7. **个人化区间** — 初始为空；游玩裁决积累后回写（把业界默认边界逐渐替换为"自己的"边界）。

## 症状 → 透镜路由

| 症状 / 任务 | 先读 |
|---|---|
| "体验枯燥 / 平" / 数值没感觉 | `flow-difficulty.md` + `core-loop.md` |
| 某机制"不够有趣" / 想变形 | `brainstorm-methods.md`（SCAMPER）+ `core-loop.md` |
| 玩家总用同一招 / 有必胜套路 | `core-loop.md` §退化策略 |
| 一局该多长 / 节奏怎么排 | `flow-difficulty.md` |
| 新手看不懂 / 教学 | `flow-difficulty.md` §教学坡道（关卡细节触发 `level-encounter.md` 建库） |
| 剧情 / 承诺-背叛的叙事感 | 触发 `narrative.md` 建库 |
| AI 对手性格 / 敌人配置 | 触发 `enemy-ai.md` / `character.md` 建库 |
| 会话开场 / 定题 | `brainstorm-methods.md`（How-Might-We） |
